# Control research first setup

> [!abstract] What this is A **closed-loop control testbed** on the LIMO. OptiTrack gives the robot's true position; a **swappable controller** (P / PID / LQR, later MPC / DeePC / Koopman) computes velocity commands; commands go straight to `/cmd_vel`. **Nav2 is bypassed** — this replaces Nav2's controller so we can compare control laws on a drift-free plant.

> [!info] Design "B" — why bypass Nav2 Nav2 = planner + controller + costmaps. This model replaces only the **controller**. No planner, no obstacle avoidance: the robot drives straight at a goal. That's intentional — obstacles would confound a controller comparison. Closing the loop on mocap means the state estimate is sub-mm and drift-free, so any tracking error is the _controller's_, not the estimator's. That's the clean testbed.

## 1. Put the control files in one folder

Make a folder, e.g. `~/matlab_limo`, and place:

|File|Role|
|---|---|
|`limo_ctrl_params.m`|Parameters + gain design (incl. toolbox-free LQR)|
|`build_limo_control_model.m`|Builds `limo_mocap_control.slx` programmatically|
|`limo_connect.m`|MATLAB↔robot connection (correct QoS + TF + status)|
|`limo_state.m`|Read all robot state|
|`limo_goal.m`|Send one goal|

Add to path once:
```matlab
addpath('~/limo/matlab/control','~/limo/matlab/goals','~/limo/matlab/perception'); savepath
```

> [!note] Why a build _script_ and not a .slx The model is generated by code (`add_block`/`set_param`). Advantage: it's diff-able, version-controllable, and every fix is reproducible. You run one script and get the model.

## 2. Understand the model before running it

Left → right signal flow:

```
/vrpn_mocap/Limo/pose ─► Subscribe ─► Bus Selector ─► quat2yaw
                                            │
                                   state = [x; y; θ]
                                            │
   goal = [xd; yd; θd] ──────────►  ┌───────────────────┐
                                    │ Variant Subsystem │   ← swap controller
                                    │  P │ PID │ LQR    │
                                    └───────────────────┘
                                            │  u = [v; ω]
                                     Saturate → E-STOP
                                            │
              Blank Twist ─► Bus Assignment ─► Publish /cmd_vel
```

> [!important] The Variant Subsystem is the whole point All three controllers share ONE interface: `in: state[3], goal[3] → out: u=[v;ω]`. Set `CTRL = 1|2|3` to swap. To add MPC/DeePC/Koopman later: add `V_MPC = Simulink.Variant('CTRL==4')`, copy a choice subsystem, replace one MATLAB Function body. Nothing else changes.

## 3. The DDS fix (one-time file, used every session)

> [!bug] Why MATLAB can't see the container's topics without this Fast DDS uses **shared memory** for same-host processes. The container runs as **root** and creates `/dev/shm/fastrtps_*` segments MATLAB (user `zakaria`) can't write. Result: MATLAB sees the _robot_ (UDP, remote) but not the _container_ (`vrpn_mocap`, Nav2). Fix: force **UDP-only**. Full detail → MATLAB_DDS_Fix.

Create `~/maps/fastdds_udp.xml` once:

xml

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<profiles xmlns="http://www.eprosima.com/XMLSchemas/fastRTPS_Profiles">
  <transport_descriptors>
    <transport_descriptor>
      <transport_id>udp_only</transport_id>
      <type>UDPv4</type>
    </transport_descriptor>
  </transport_descriptors>
  <participant profile_name="udp_participant" is_default_profile="true">
    <rtps>
      <userTransports><transport_id>udp_only</transport_id></userTransports>
      <useBuiltinTransports>false</useBuiltinTransports>
    </rtps>
  </participant>
</profiles>
```

Make it permanent so you stop doing it by hand:

bash

```bash
# container ~/.bashrc
echo 'export FASTRTPS_DEFAULT_PROFILES_FILE=/root/maps/fastdds_udp.xml' >> /root/.bashrc
```

matlab

```matlab
% MATLAB startup.m  (create in ~/Documents/MATLAB/)
setenv("FASTRTPS_DEFAULT_PROFILES_FILE", fullfile(getenv('HOME'), "maps", "fastdds_udp.xml"));
```

## 4. Build the model once

matlab

```matlab
limo_ctrl_params            % loads P, CTRL, variant objects; prints LQR gain K
build_limo_control_model    % builds + opens limo_mocap_control.slx
```

> [!note] LQR gain design `limo_ctrl_params` designs the gain with `K = lqr(A, B, Q, R)` (Control System Toolbox) on the body-frame unicycle error dynamics. Gave `K = [-2 0 0; 0 -4 -2.4495]`. It's baked into the LQR block automatically at build time — no manual pasting.
> 
> Tune the trade-off via `Q` (state penalty) and `R` (effort penalty) in the params file, then rebuild. Bigger `Q` → tighter tracking, more aggressive; bigger `R` → gentler, less effort.
> 
> _(Historical note: before the toolbox was installed, this was solved manually via a Hamiltonian eigendecomposition of `H = [A, -B R⁻¹ B'; -Q, -A']`. Same result. Kept here only as background.)_

> [!warning] Programmatic build is release-sensitive If the script warns it couldn't set a block parameter, set that one field by hand in the block dialog. The known-tricky ones (already handled): QoS params are UPPERCASE (`QOSReliability`), Subscribe **port 2** is the message bus (port 1 is `IsNew`), and bus signals must be assigned **after** an Update Model.

## 6. First smoke test (robot must NOT move)

1. Robot drivers + `vrpn_mocap` running (see daily note). **Nav2 stopped.**
2. Press **Run**.
3. The robot should **not** move — the E-STOP Manual Switches start in the **zero** position.
4. **Push the robot by hand** → the `x`, `y`, `theta` displays should track it.

> [!success] If the displays track the robot Mocap → Simulink is closed. Sensing works. You're one double-click (the E-STOPs) away from live control. Do the first _moving_ run slow — see the daily note's safety section.

---

## 7. Safety model (understand before enabling motion)

- **Two E-STOP Manual Switches** feed zero-velocity by default. Motion only happens when you _double-click_ them during simulation. Double-click again → instant stop.
- **Saturation blocks** clamp `v ≤ P.v_max`, `ω ≤ P.w_max`. First runs: `P.v_max = 0.15`.
- **Only one thing may publish `/cmd_vel`.** Stop Nav2 before running this model, or they fight over the robot.

## 8. Analysis tools (Control System Toolbox) — for comparing controllers

Once you're comparing P / PID / LQR (and later MPC / DeePC), don't just eyeball the XY trace — quantify it. The toolbox gives you:

|Function|Use|
|---|---|
|`step`, `impulse`|time-domain response of a linearized loop|
|`bode`, `nyquist`|frequency response|
|`margin`|gain/phase stability margins|
|`pole`, `eig`, `damp`|closed-loop poles, natural modes, damping|
|`care`, `dare`|Riccati solvers for custom optimal control|

These connect directly to the eigenvalue / natural-mode / stability analysis in the control-theory notes — the linearized error dynamics `A`, `B` in `limo_ctrl_params.m` are exactly what you feed to `ss(A,B,C,D)` then `step`/`bode`/`pole`.

> [!tip] Fair controller comparison = same plant, same metric Because every controller honors the same interface and runs on the same drift-free plant, you can compare them on identical metrics: settling time (`step`), overshoot, RMSE-to-goal, control effort (∫|u|). Record each run's `/vrpn_mocap/Limo/pose` and compute offline — same as the nav comparison experiment.
